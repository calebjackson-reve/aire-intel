## Trigger

Weekly cron every Friday at 7:00 AM CT. Compiles a read-only digest of Baton Rouge agent market activity for the week.

## Input

- Paragon MLS via `src/lib/paragon.ts → fetchListings()` — new listings and status changes in last 7 days across Baton Rouge target ZIPs (70808, 70810, 70816, 70820, plus configurable list)
- `Setting["competitor.trackedZips"]` — JSON array of ZIP codes to monitor (default: Caleb's corridors)
- `Setting["competitor.trackedAgentNames"]` — JSON array of competing agent names to highlight (optional; used for name-matching in listing agent fields)
- `ZillowHotListing` — viral listings from this week (already fetched by market-intel agent)
- Recent `ScheduledPost` data for Caleb (to compare his posting frequency against market activity level)

## Actions

1. Pull all new MLS listings + status changes from last 7 days in tracked ZIPs
2. Aggregate by listing agent:
   - Count of new listings per agent
   - Count of under-contract / sold per agent
   - Average list price for new listings
3. Identify notable activity:
   - Any agent with ≥ 5 new listings this week (high volume)
   - Any listing that went under contract in < 3 days (hot property / competitive market signal)
   - Price reductions > 5% on any active listing (motivated seller)
   - Listings in Caleb's target corridors that match active buyer searches
4. Generate a plain-text weekly digest via Claude Haiku (max 300 words): "This week in Baton Rouge RE: [3–5 notable observations]. [1 opportunity or market insight for Caleb]."
5. Write digest to `DailyBrief` for Friday's brief
6. Create `Notification`: "Weekly market digest ready — [N] notable signals in your corridors"
7. If any listing matches an active `BuyerSearch` that was NOT already caught by loop #02 (listing-alert-buyer-match): enqueue a buyer alert draft

## Oracle

**What external source of truth grades the output?**  
This is a monitoring loop — no actionable oracle. The value is informational. Secondary oracle: if Caleb reads/acts on the digest (measured by `DailyBrief` section being opened or a buyer alert being approved from this loop's output).

**Acceptance threshold:**  
Paragon returns ≥ 5 listing changes per week in tracked ZIPs (indicating market data is flowing). Digest generated without API errors.

**Rejection signal:**  
Paragon API returns error or 0 results for 2 consecutive weeks → log and alert: "Paragon data may be stale — check PARAGON_API_KEY."

## Memory

- `Setting["competitor.trackedZips"]` — configurable ZIP list
- `Setting["competitor.lastDigest"]` — date of last digest (within-6-days guard)
- `ZillowHotListing` — reuses data already fetched by market-intel agent (no duplicate API call)

## Surface

- `DailyBrief` — competitor/market digest section (Friday)
- Dashboard `Notification` (Friday morning)
- The digest is informational — no `ActionQueue` items are created by this loop except for buyer match alerts that slipped through loop #02

## Safety Rails

- **Human chokepoint:** Read-only analysis. No outbound actions generated by this loop directly (except buyer match dedup check). Caleb reads the digest and decides whether to act.
- **Blast radius:** Zero writes to lead data, ContactLog, or ActionQueue (except the buyer match dedup case). Pure read + Notification + DailyBrief write.
- **Rate limit / cap:** Once per week. Paragon API calls bounded by the number of ZIPs (one call per ZIP, max 10 ZIPs).
- **Idempotency:** `Setting["competitor.lastDigest"]` within-6-days guard.
- **Exit condition:** `PARAGON_API_URL` not set → skip and note: "Paragon not configured — competitor monitor inactive." `Setting["loop.competitor_monitor.disabled"] = "true"` to pause.
